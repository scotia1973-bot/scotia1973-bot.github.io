Why Email Verification Matters for SaaS

Here's what happens when you don't verify emails at signup:

  • 40-60% bounce rates on your first campaign (users typed .con instead of .com)
  • Fake account proliferation — disposable emails (@mailinator.com, @10minutemail.com) used to game free trials
  • Sender reputation damage — ISPs see high bounce rates and start flagging all your mail as spam
  • Wasted infrastructure costs — storing, processing, and eventually trying to email invalid addresses

Most developers think they need to send a confirmation email (the "click to verify" flow). That helps, but it's not enough. Confirmation emails bounce too. The real solution is syntactic + MX-level validation at the form level — checking that the email format is valid and that the domain actually accepts mail — before the user hits Submit.

Step-by-Step Integration

1 Add the API call to your form handler

First, add a simple async validation function. This works with any framework — React, Vue, vanilla JS, or your backend:

async function verifyEmail(email) {
  const url = `https://email-verify.scotia1973.workers.dev/verify?email=${encodeURIComponent(email)}`;
  const response = await fetch(url);
  return response.json();
}

2 Validate on form submit (or on blur)

document.getElementById('signup-form')
  .addEventListener('submit', async (e) => {
    e.preventDefault();
    const email = document.getElementById('email').value;
    const result = await verifyEmail(email);

    if (result.valid) {
      // Email is good — submit the form
      form.submit();
    } else {
      showError(`Invalid email: ${result.reason}`);
    }
  });

3 Show user-friendly error messages

Instead of a generic "invalid email" error, give your users specific feedback based on what the API returns:

function showError(reason) {
  const messages = {
    'invalid_format': 'Please enter a valid email address (e.g., name@domain.com)',
    'no_mx_record': 'This domain does not accept email. Check for typos.',
    'disposable_email': 'Please use a permanent email address, not a temporary one.',
    'role_based': 'Please use a personal email address, not a role-based one.',
    'smtp_rejected': 'This mailbox does not exist. Double-check the address.'
  };
  document.getElementById('error-msg').textContent =
    messages[reason] || 'Invalid email address. Please try again.';
}

Pro tip: Call the API on the blur event (when the user tabs away from the email field), not just on submit. This gives real-time feedback and feels more polished.

Handling Rate Limits Gracefully

The free tier allows 50 verifications per day — plenty for most signup forms on early-stage SaaS products. Here's how to implement a simple fallback:

async function verifyEmail(email) {
  try {
    const url = `https://email-verify.scotia1973.workers.dev/verify?email=${encodeURIComponent(email)}`;
    const response = await fetch(url);
    if (response.status === 429) {
      // Rate limited — fall back to basic regex validation
      return { valid: /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email) };
    }
    return response.json();
  } catch {
    return { valid: true }; // Fail open if API is unreachable
  }
}

What About Backend Validation?

Frontend-only validation is convenient, but always validate on the server too. Here's a Node.js/Express example:

const express = require('express');
const app = express();
app.use(express.json());

app.post('/signup', async (req, res) => {
  const { email } = req.body;

  const response = await fetch(
    `https://email-verify.scotia1973.workers.dev/verify?email=${encodeURIComponent(email)}`
  );
  const result = await response.json();

  if (!result.valid) {
    return res.status(400).json({
      error: 'Invalid email',
      reason: result.reason
    });
  }
  // Create user...
});
