1. Introduction

2025-2026 has been the era of AI agents — autonomous systems that can reason, plan, and take actions on behalf of users. But for an AI to actually do something (generate a QR code, check email validity, convert a color, look up an IP address), it needs a bridge between its language understanding and real-world APIs.

That bridge is the Model Context Protocol (MCP). MCP is an open standard developed by Anthropic that defines how AI assistants connect to external tools, APIs, and data sources. Think of it as USB-C for AI — a universal connector that lets any AI assistant plug into any service.

In this complete guide, you'll learn what MCP servers are, how they work, how to build one, and how the GadgetHumans API Hub MCP Server serves as a real-world example.

2. What Is the Model Context Protocol?

MCP (Model Context Protocol) is a standardized protocol that allows AI language models to interact with external tools and data sources. It defines three core primitives:

  • Tools: Function-callable operations the AI can invoke (e.g., "generate QR code", "validate email")
  • Resources: Structured data the AI can read (e.g., documentation, user profiles)
  • Prompts: Pre-built prompt templates for common tasks

Before MCP, every AI integration was custom-built — a bespoke "tool use" implementation for each AI platform and each API. MCP standardizes this: one MCP server can serve Claude, Gemini, GPT, open-source models, and any other MCP-compatible client.

Key Benefits of MCP

  • Standardization: One protocol, any AI assistant. No vendor lock-in.
  • Discoverability: The AI can ask the server "what tools do you have?" and get a list with descriptions.
  • Type Safety: Tool parameters are defined with JSON Schema — the AI validates inputs before calling.
  • Bidirectional: The server can push resource updates to the client, enabling real-time data.
  • Transport Agnostic: MCP works over stdio (local processes) or SSE (HTTP streaming).

For the full specification, visit the official MCP documentation.

3. MCP Architecture

MCP uses a client-server architecture:

┌─────────────┐     JSON-RPC      ┌──────────────────┐
│  AI Assistant │ ◄───────────────► │    MCP Server    │
│  (Claude,     │    MCP Protocol   │  (Tools, Data,   │
│   Gemini...)  │                   │   Prompts)        │
└─────────────┘                     └──────────────────┘
                                              │
                                              ▼
                                    ┌──────────────────┐
                                    │  REST APIs / DBs  │
                                    │  / File System    │
                                    └──────────────────┘

MCP Client (Host): The AI assistant or application that connects to MCP servers. It discovers available tools and resources, and executes tool calls on behalf of the user.

MCP Server: A lightweight process that exposes tools (functions), resources (data), and prompts (templates). Multiple MCP servers can be connected to a single AI assistant — each server provides a different domain of capabilities.

Transport Layer: Two options:

  • stdio — The server runs as a subprocess of the client, communicating via stdin/stdout. Ideal for local development.
  • SSE (Server-Sent Events) — The server runs as an HTTP process, communicating via POST requests and SSE streams. Ideal for remote servers.

4. How MCP Servers Work

When an AI assistant needs to use a tool, the flow is:

  1. Initialization: The client connects to the MCP server and receives a list of available tools with their schemas.
  2. Intent Recognition: The AI analyzes the user's request and determines which tool to call, with which parameters.
  3. Tool Call: The client sends a tools/call JSON-RPC request to the MCP server with the tool name and arguments.
  4. Execution: The MCP server executes the tool (e.g., calling a REST API, querying a database, reading a file).
  5. Response: The server returns the result to the client, which the AI incorporates into its response to the user.
// Example: MCP Tool Call (JSON-RPC)
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "generate_password",
    "arguments": {
      "length": 20,
      "symbols": true,
      "numbers": true
    }
  }
}

// Response:
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      { "type": "text", "text": "X7#kL9$mN2@pQ4!rT8&" }
    ],
    "isError": false
  }
}

6. Building Your Own MCP Server

Building an MCP server is straightforward with the official SDKs. Here's a minimal example in Python:

from mcp.server import Server, NotificationOptions
from mcp.server.models import InitializationOptions
import mcp.server.stdio

# Create an MCP server
server = Server("my-tools")

# Register a tool
@server.list_tools()
async def handle_list_tools():
    return [
        {
            "name": "hello_world",
            "description": "A simple greeting tool",
            "inputSchema": {
                "type": "object",
                "properties": {
                    "name": {
                        "type": "string",
                        "description": "Name to greet"
                    }
                },
                "required": ["name"]
            }
        }
    ]

@server.call_tool()
async def handle_call_tool(name: str, arguments: dict):
    if name == "hello_world":
        greeting = f"Hello, {arguments['name']}!"
        return [{"type": "text", "text": greeting}]
    raise ValueError(f"Unknown tool: {name}")

# Run the server over stdio
async def main():
    async with mcp.server.stdio.stdio_server() as (read_stream, write_stream):
        await server.run(
            read_stream,
            write_stream,
            InitializationOptions(
                server_name="my-tools",
                server_version="1.0.0"
            )
        )

if __name__ == "__main__":
    import asyncio
    asyncio.run(main())

To wrap an existing REST API as MCP tools, structure each API endpoint as a tool function that makes an HTTP request and returns the result. The GadgetHumans MCP server follows this exact pattern — each tool wraps a call to api.gadgethumans.com.

Generic MCP Client

import { Client } from "@modelcontextprotocol/sdk/client/index.js";
import { StdioClientTransport } from "@modelcontextprotocol/sdk/client/stdio.js";

const transport = new StdioClientTransport({
  command: "npx",
  args: ["-y", "@scotia1973-bot/api-hub-mcp"]
});

const client = new Client({
  name: "my-app",
  version: "1.0.0"
});

await client.connect(transport);

// List available tools
const tools = await client.listTools();
console.log("Available tools:", tools);

// Call a tool
const result = await client.callTool({
  name: "generate_password",
  arguments: { length: 16, symbols: true }
});
console.log("Generated password:", result.content);

8. MCP Server Use Cases

MCP servers are useful anywhere you want AI assistants to interact with real systems:

  • Developer Utilities: The GadgetHumans pattern — wrap developer APIs (QR codes, passwords, JSON tools) so AI can use them naturally via conversation.
  • Database Access: Let AI query your database with natural language — "Show me all users who signed up last month."
  • File System Operations: Give AI read/write access to specific directories for code generation or document processing.
  • GitHub/Git Integration: MCP servers for creating repos, filing issues, and reviewing PRs from within the AI chat.
  • Internal APIs: Expose company-internal APIs (CRM, project management, ticketing) as MCP tools for AI-powered workflows.
  • Web Scraping & Search: Let AI fetch and analyze web content, search documentation, or crawl APIs.

The common thread: any task that a developer would normally do by writing code, calling an API, or running a command can become an MCP tool that an AI assistant can invoke conversationally.
